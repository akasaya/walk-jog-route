## 1. アプリケーションセキュリティ

### 入力バリデーション（最重要）

外部入力（ユーザー入力・API レスポンス・URL パラメータ）はすべてバリデーション後に使う。

```python
# FastAPI / Pydantic でスキーマレベルのバリデーションを使う
class RouteRequest(BaseModel):
    lat: float = Field(ge=-90, le=90)     # 緯度: -90〜90
    lon: float = Field(ge=-180, le=180)   # 経度: -180〜180
    distance_km: float = Field(gt=0, le=100)
```

- `lat` / `lon` は数値範囲チェック必須（`422` を返す）
- 文字列フィールドは最大長を設定する
- バリデーションなしで ORM や外部 API にそのまま渡さない

### 認証・認可

- JWT を使う場合: `exp` クレームを必ず検証し、署名アルゴリズムは `RS256` または `ES256` を使う（`none` アルゴリズムは禁止）
- エンドポイントに認証が必要かどうかを設計時に明示し、実装時に省略しない
- 認可チェック: 他ユーザーのリソース（ルート履歴等）へのアクセスは `userId` 検証後に許可する

### XSS / インジェクション防止

- HTML を返す箇所は出力を必ずエスケープする。React Native / Web ではフレームワーク外から生 HTML を DOM に直接挿入する API を使わない
- DynamoDB の `FilterExpression` は `boto3` のパラメータバインディングを使い、文字列連結で条件を作らない
- SQL を使う場合（将来）: プリペアドステートメント必須

### レートリミット

- `/routes/suggest` など計算コストの高いエンドポイントにはリクエスト制限を設ける
- 実装例: `slowapi`（FastAPI）でユーザー単位または IP 単位に制限

## 2. データ保護（位置情報 = PII）

walk-jog-route はユーザーの現在地（GPS 座標）を扱う。座標は個人の行動履歴に直結するため **PII（個人識別可能情報）として扱う**。

### ログ出力のルール

```python
# NG: 座標をそのままログに出す
logger.info(f"Route request: lat={lat}, lon={lon}")

# OK: 精度を落とす・または省略する
logger.info(f"Route request: area={round(lat, 1)},{round(lon, 1)}, dist={distance_km}km")
```

- アクセスログ・アプリログに生の座標（小数点 4 桁以上）を含めない
- エラーログに `request body` をそのまま出力しない（位置情報が含まれる）
- CloudWatch Logs の保持期間は必要最小限に設定する

### データ収集・保持

- 収集するデータは機能に必要な最小限にとどめる
- ルート履歴の保持期間をユーザーに明示し、期限後は削除する
- バックアップ（DynamoDB PITR）にも同じ保持ポリシーを適用する

### 暗号化

- **転送中**: HTTPS 必須（App Runner + CloudFront はデフォルトで TLS 終端）
- **保存時**: DynamoDB は AWS 管理キーで暗号化（デフォルト有効）。追加要件がある場合は CMK（Customer Managed Key）を使う

## 3. 秘密情報の開発時管理

`50-agent-safety.md` のハードコード禁止ルールを補完する、開発ワークフロー上の指針。

### ローカル環境

```
.env.local          ← 個人の開発用。必ず .gitignore に含める
.env.example        ← キー名のみ記載（値なし）。コミット可
```

- `.env.local` は他者と共有しない（Slack・PR コメントへの貼り付け禁止）
- API キーが漏洩した可能性がある場合は即座に失効させる

### テストコードのルール

```python
# NG: 実在する個人の座標を使う
def test_suggest_route():
    lat, lon = 35.6894, 139.6917  # 実際の場所の特定可能な座標

# OK: 意味のない架空の座標またはダミー定数を使う
TEST_LAT, TEST_LON = 35.0, 139.0  # 丸め値・架空の値
```

- テストコードに本物の API キーを書かない
- CI の環境変数（GitHub Secrets）にテスト用の別キーを用意する

## 4. 依存関係のセキュリティ

### 脆弱性スキャン

```bash
# Python
pip-audit                    # 既知脆弱性チェック
pip install --upgrade <pkg>  # 脆弱性のあるパッケージを更新

# Node.js / pnpm
pnpm audit
pnpm audit --fix
```

- PR マージ前に `pip-audit` / `pnpm audit` を実行し、HIGH・CRITICAL レベルの脆弱性がある状態でマージしない
- GitHub Dependabot を有効化し、自動 PR で依存関係を更新する

### バージョン管理

- `requirements.txt` はバージョンを固定する（`fastapi==0.115.0`、`>=` のみは不可）
- `package.json` は `pnpm-lock.yaml` とともに管理する
- `latest` タグのコンテナイメージを本番に使わない（`40-aws-cicd.md` 参照）

## 5. セキュリティテスト

`30-testing.md` のテスト方針を補完する、セキュリティ観点の追加要件。

### 必須テストケース（route-core / API）

- 緯度・経度の範囲外値 → 422 を返すこと
- 認証トークンなしでの保護エンドポイントアクセス → 401 を返すこと
- 他ユーザーのリソース ID を使ったアクセス → 403 を返すこと
- 極端に大きなリクエストボディ → 適切にリジェクトされること

### 動的テスト（MVP 後）

- OWASP ZAP によるスキャンを CI に組み込む（ステージング環境対象）
- ペネトレーションテストは公開前・大規模な仕様変更後に実施を検討する

---

## エージェントへのルール

- 位置情報（lat / lon）をログ・エラーメッセージ・レスポンスに生の精度で含めない
- テストデータには実在する個人の座標・API キーを使わない
- 入力バリデーションなしで外部データを DynamoDB・外部 API に渡さない
- 新しい外部ライブラリを追加する場合、`pip-audit` / `pnpm audit` の通過を確認してからコミットする
- 認証・認可の実装を「あとで追加」として省略しない。設計時に決定し、エンドポイント実装と同時に追加する
