# Testing Rule

このプロジェクト（walk-jog-route）のテスト方針を定義します。

## スタック

| レイヤー | フレームワーク | ランナー |
|----------|---------------|---------|
| Python backend | pytest + httpx | `pytest` |
| Web frontend | Vitest + Testing Library | `vitest` |

---

## テスト優先順位

```
1. route-core（ルート生成ロジック） ← 最重要
2. backend API（FastAPI エンドポイント）
3. Web フロントエンド（コンポーネント）
```

---

## 1. route-core（ルート生成ロジック）

純粋関数が多いためモック不要。実アルゴリズムをそのままテストする。

### テスト観点

#### 距離・座標計算
- 2点間距離の精度（ハーバーサイン公式: ±1% 以内）
- 緯度経度の境界値: `(0, 0)`, `(90, 180)`, `(-90, -180)`
- 同一座標を渡したとき距離 = 0

#### ルート生成
- 指定距離に対する収束精度: ±10% 以内
- 最小ケース: 0.5 km
- 最大ケース: 50 km
- ループ判定: スタート地点に戻るか
- 重複経路: 同じ道を2度通らない（または許容回数を超えない）
- 袋小路・行き止まりに入り込まないこと

#### 異常系
- 海上・建物内など経路不能な座標を起点にした場合のエラー返却
- グラフデータ欠損時のフォールバック動作
- ルートが見つからない場合は例外ではなく `None` または空リストを返す（仕様に合わせる）

### ファイル配置
```
tests/
  route_core/
    test_distance.py
    test_route_generator.py
    test_waypoint_selector.py
```

## 2. backend API（FastAPI）

`TestClient`（httpx ベース）を使い、DB を含む統合テストを行う。

### テスト観点

#### 正常系
- `POST /routes/suggest` で有効な緯度経度・距離を渡すと 200 とポリラインを返す
- レスポンス形式が固定されている（`polyline`, `distance_m`, `estimated_minutes`）

#### 入力バリデーション
- `lat` が範囲外（< -90, > 90）→ 422
- `lon` が範囲外（< -180, > 180）→ 422
- `distance_km` が 0 以下 → 422
- `distance_km` が上限超過（例: 100 km 超）→ 422 または 400

#### 異常系
- 外部マップAPI がタイムアウト → 503 を返す（外部 API はモック可）
- 内部エラー → 500 で詳細をレスポンスに含めない

### ルール
- 外部の地図タイル・ジオコーダ API はモックしてよい
- DB（将来実装）はテスト用 DB に接続し、モックしない
- 各テストは独立して実行できること（共有状態を持たない）

### ファイル配置
```
tests/
  api/
    test_routes.py
    test_health.py
```

---

## 3. Web フロントエンド（Vitest + Testing Library）

UI の細部より「ユーザーが操作できる」ことを確認するテストを書く。

### テスト観点

#### コンポーネント
- 距離スライダー: 値変更 → state 反映
- ルート表示カード: `polyline` が渡されたとき地図上に描画される（地図ライブラリはモック）
- エラー表示: API エラー時にメッセージが表示される

#### 統合（ページレベル）
- 現在地取得 → ルート提案ボタン押下 → ルート表示までのフロー
- Geolocation API はモックする（`navigator.geolocation`）

### ルール
- Google Maps / Leaflet などの地図ライブラリはモックしてよい
- スナップショットテストは使わない（変更コストが高い）
- `data-testid` は最終手段。`role` / `label` / `placeholder` を優先する

### ファイル配置
```
src/
  components/
    RouteCard/
      RouteCard.test.tsx
  pages/
    Home.test.tsx
```

## カバレッジ目標

| スコープ | 目標 |
|---------|------|
| route-core | 90% 以上 |
| API エンドポイント | 80% 以上 |
| Web コンポーネント | 70% 以上 |

カバレッジは目安。数値を上げるためだけのテストは書かない。

---

## エージェントへのルール

- **route-core のロジックはモック禁止**。純粋関数なのでそのまま呼び出す。
- **外部 API（地図タイル、ジオコーダ）はモック可**。ただしモック時はリアルな形式のレスポンスを返すこと。
- テストは 1 ケース 1 アサーションを基本とする（複数 assert は関連するものに限る）。
- `parametrize`（pytest）/ `test.each`（Vitest）で境界値・異常値をまとめる。
- テスト関数名は `test_<対象>_<状態>_<期待結果>` 形式にする。
  - 例: `test_calculate_distance_same_point_returns_zero`
- CI（GitHub Actions）でテストが通ることを必須とする。ローカルのみ通過は不可。
